Priority Update Registry

A shared on-chain registry for block-scoped state updates, designed for builder integration.

Problem

Many contracts depend on fresh state that must be updated every block: price feeds, oracle data.

Constraints:

  • Updates should only land when they are actually used.
  • Updates must land before the transactions that use them. The contract needs fresh state at the time it is called.

This means the block builder needs to figure out, during simulation, that a transaction needs an update, and then place that update in front of it.

Proposed Solution

One canonical registry contract where all priority updates live.

  • Standardized: Builders learn one interface
  • Composable: Multiple updates from different protocols can be batched into a single transaction
  • Block-scoped: Updates are only valid for the current block (enforced by block timestamp), automatically stale after
  • Predictable storage: Fixed layout means builders can apply updates directly to state without executing arbitrary code
  • No execution risk: The registry only writes to its own storage. No frontrunning, no complex execution

Interface

Each target's state is a list of slots: the first slot holds 8 bytes, followed by k additional 32-byte slots.

The registry exposes two core operations:

Updating state.

updateState(address target, uint256 blockTimestamp, uint256[] slots)

An authorized updater calls this to write fresh state for a target contract. The blockTimestamp must match the current block, ensuring the data is never stale.

batchUpdateStateWithSignature(SignedUpdate[] updates)

For batching multiple updates, accepts an array of EIP-712 signed updates that anyone can submit on behalf of updaters.

Reading state.

getState(uint256 numSlots) returns (uint256[] memory)

A target contract calls this on the registry (msg.sender determines which target's state is returned). Reverts with StateNotUpdated if no update has been written in the current block, so consumers never read stale data.

How Updaters Submit Updates to the Builder

Updaters sign priority updates and send them to the builder. The builder keeps a pool of pending updates, one per target contract (latest wins).

Updates arrive via eth_sendBundle. The builder controls inclusion and never lets update transactions land on their own.

How the Builder Includes Updates in a Block

When the builder simulates a user transaction, it watches for calls to the registry. If a contract reads from the registry, the builder inserts the update transaction before the user transaction.

In this example, Contract X uses the registry for fresh price data.

Final block ordering after the builder inserts the update:

Block
tx 0: ...
tx 1: ...
tx 2: Update Contract X (inserted by builder, writes fresh state to registry)
tx 3: User Transaction — Contract A → Contract X → Registry (reads fresh state ✓)
tx 4: ...
The builder does not need to execute arbitrary update logic. Because the registry has a fixed storage layout, the builder can apply the update directly to state. It just writes the known values to the known slots. The update transaction is then placed before the user transaction so on-chain execution matches the pre-applied state.

Batching

When multiple contracts use updates, the builder can batch all of them into a single transaction at the top of the block. Since update transactions only write to the registry and have no dependencies on other state, they can always go first.

Block
tx 0: Batched update for Contract X and Contract Y (single tx, top of block)
tx 1: ...
tx 2: User tx — calls Contract X (reads fresh state ✓)
tx 3: ...
tx 4: User tx — calls Contract Y (reads fresh state ✓)
tx 5: ...

Summary

  1. Builder starts simulating a user transaction
  2. Deep in the callstack, some contract calls Registry.getState() for Contract X
  3. Builder intercepts this storage read and checks: is there a pending update for Contract X?
  4. If yes, builder applies the update values directly to registry storage slots (no tx execution needed for simulation)
  5. Simulation continues with fresh state, getState() returns the updated values
  6. After simulation succeeds, builder inserts the actual update transaction before the user transaction in the block
  7. On-chain execution replays the same sequence: update tx writes fresh state, then user tx reads it

Costs

Each contract gets 8 bytes plus k additional 32-byte slots of state. Gas costs scale linearly with k.

OperationGas
Direct update21000 + 6876 + k × 5212
Batched update (n updates)21000 + 830 + n × (13331 + k × 5236)
Read (cold)3090 + k × 2269
Read (warm)1090 + k × 269
